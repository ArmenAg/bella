# Current Records Seed Audit

This audit documents the reusable current-records seed saved at:

```text
supabase/seed/005_current_records_bootstrap.sql
```

The seed is generated by:

```bash
npm run bootstrap:seed-sql
```

It is deterministic and safe to rerun. It imports into the local demo profile `bella.demo@example.test` unless the generator is called with a different profile email.

## Input Coverage

The current bootstrap scanner covers:

- `../records_md`
- `../reports_md`
- `../Fw_ Medical Files`
- `../HealthSummary_May_07_2026/1 of 1 - My Health Summary.PDF`
- Top-level importable workspace files such as Stanford/Overlake PDFs

The scan registers every importable file with a stable source ID, hash, original workspace path, source-library metadata, extraction status, and review status.

Latest generation summary:

- 271 source rows scanned
- 116 accepted source rows emitted into the seed
- 69 accepted timeline/procedure/imaging events emitted into the seed
- 7 duplicate or unsafe event candidates rejected
- 183 attachment candidates identified but not storage-backed yet
- 2 narrative entry candidates identified but not auto-split yet

When applied on top of earlier seeds, the active count of `bootstrap_import` sources can be lower than 116 because `005_current_records_bootstrap.sql` de-duplicates by `citation` and maps events to an existing canonical source row when one is already present.

## Source-Only Decisions

These files are intentionally kept as source-library rows but not imported as duplicate timeline events:

- `records_md/generated/user_provided/Stanford_Tawfik_2025_10_06_User_Provided.md`
- `records_md/generated/user_provided/Stanford_MyHealth_Tawfik_2026_01_26.md`
- `records_md/generated/stanford_pdfs/Tawfik_1_26.md`
- `records_md/generated/stanford_pdfs/Bella_Letter_2_9_Tawfik.md`
- `records_md/generated/stanford_pdfs/Bella_Stanford_Surgery_Consult_29.md`

These narrative logs are source-only until they are manually split into structured pain/log entries:

- `records_md/generated/user_provided/Pain_Log_User_Provided.md`
- `records_md/generated/user_provided/Unknown_Date_Scar_Injection_Response_Log.md`

The pain logs contain important material, but the date/year boundaries and procedure linkage are ambiguous enough that importing them automatically would create false precision.

## Raw PDF Metadata

Raw PDF metadata is curated by path in `scripts/historical-import/scan-sources.mjs` so source-library rows do not inherit misleading dates from DOBs, footers, or MyChart page chrome.

Examples:

- `Uritskiy post 10.17 scar injection.pdf` is dated 2025-10-24, not Bella's DOB or the MyChart footer date.
- `Tawfik 1:26.pdf` is dated 2026-01-26.
- `HealthSummary_May_07_2026/1 of 1 - My Health Summary.PDF` is labeled as an Overlake MyChart health summary export dated 2026-05-07.

## Known Gaps

- Attachment candidates are discovered but not uploaded to Supabase Storage yet. The seed does not create attachment rows for files without storage paths.
- Narrative pain logs are not split into structured pain entries yet.
- Raw DICOM and image directories are represented by generated imaging reports and attachment candidates, not by storage-backed binary records.
- The seed is PHI-bearing and should stay in a private repository/environment.
